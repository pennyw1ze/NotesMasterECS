A1. INFORMATION LEAKING IN THE SALTED-HASH SELECTIVE DISCLOSURE
    SOURCE: [YOU] - "EUDI Wallet.md", section "Problemi che trovo io"
    Description: Proving a predicate such as "I am an adult" with salted hashes
    (and no ZKP) forces disclosure of the WHOLE field, i.e. the full date of
    birth. Likewise proving nationality may leak the full address / city of
    residence. The selective-disclosure mechanism lacks granularity: it can only
    reveal or hide a whole attribute, never a derived predicate.
    Note on the proposed mitigation: the spec suggests the issuer pre-bakes
    boolean fields (e.g. "adult: yes/no"). You correctly reject this as
    unworkable: it does not scale to arbitrary thresholds (over 19, over 20, ...).
    Real fix: Zero-Knowledge Proofs.

A2. IDENTITY/ATTRIBUTE LINKABILITY CHAIN VIA DEVICE BINDING
    SOURCE: [YOU/PAPER] - "EUDI Wallet.md", section "User leaking"
    (you note the base linkability problem was already known and NOT first found
     by you; the cloning/impersonation reasoning below is your own extension)
    Description: When an attestation is presented, a device-binding challenge
    ties the selectively disclosed attributes to the device. The device is tied
    to the PID, and the PID reveals identity. Therefore the chain
    selectively-disclosed attribute -> device -> PID -> identity lets a verifier
    (or colluding parties) collapse anonymity:
    once identity is revealed it can be linked back to the device and hence to
    all previously "anonymous" disclosed attributes -> privacy vanishes.
    The "present only once" and time-limit countermeasures do not prevent this,
    as the linking action can be done quickly.
    Acknowledged by the spec itself: the "only viable mitigation" is ZKPs.

A3. POSSIBLE USE OF ANOTHER PERSON'S DOCUMENT / WEAK BINDING OF PREDICATE TO HOLDER
    SOURCE: [YOU] - "EUDI Wallet.md", section "User leaking"
    Description: When only a predicate / partial field is shown (e.g. birth-date
    field of a PID, signed by e.g. the Italian government), there is no strong
    proof that the disclosed birth date is bound to the presenter's own identity.
    In the current implementation one could load another person's document into
    the wallet and authenticate "as" them for such predicate checks. You flag
    this as a problem that should not be possible.

A4. PSEUDONYMS ARE FUNCTIONALLY USELESS / STILL LINKABLE
    SOURCE: [YOU] - "EUDI Wallet.md" (Conclusioni 26/03/2026 + pseudonym note)
    Description: In the current architecture pseudonyms are nothing more than a
    locally generated key pair used for authentication; they use no special
    technology and add little. Because a pseudonym must be presented together
    with an attestation at least the first time (or be provided by a trusted
    third party to grant privileges), it remains a possible victim of a linking
    attack. Conclusion: pseudonyms as currently specified are nearly pointless.

A5. NO BACKUP / RECOVERY MECHANISM (LOSS OF WALLET = LOSS OF IDENTITY)
    SOURCE: [YOU] - "EUDI Wallet.md" (final notes) and FINDINGS 04/2026
    Description: The system provides no backup/recovery. If the wallet device is
    lost, the user loses access. Keys are trapped inside the Secure Element and
    cannot be moved out (QESRC keys cannot leave the SE). You propose MPC-based
    social recovery (Shamir secret sharing among trustees) as a direction.

A6. SECURE ELEMENT AS A SINGLE POINT OF FAILURE (SPOF)
    SOURCE: [YOU/PAPER] - "EUDI Wallet.md" FINDINGS + Frigo/shelat report
    Description: All security rests on a single hardware Secure Element. There is
    no multi-party computation to remove this single point of failure, neither in
    the ARF nor in the Longfellow (Frigo et al.) implementation. You propose
    "de-materialising" the SE into a distributed MPC/2PC computation (between the
    device and a second device or trusted server) to remove the SPOF and reach
    100% device coverage instead of ~10.5%.

A7. POST-QUANTUM INSECURITY OF THE DEVICE-BINDING / SE SIGNATURE LAYER
    SOURCE: [YOU/PAPER] - "EUDI Wallet.md" FINDINGS (citing Lehmann et al.,
            "Vision") + Frigo/shelat report
    Description: Current Secure Elements only support ECDSA over P-256. The
    hardware cannot natively compute advanced (BBS) signatures or native ZKPs.
    Because device binding mandatorily involves the SE, the binding signature is
    stuck on legacy, quantum-vulnerable ECDSA. Bridging to anonymous/PQ schemes
    requires heavy external ZKPs (ZKAttest / zkSNARK), making the system slower.
    REFINEMENT (your own analysis, "Thesis advance 02-05-2026.md"):
    The device-binding key does NOT actually need to be PQ-secure, because it is
    never revealed - only a ZK proof of possession (signing a nonce) is shown.
    Even if a quantum computer recovered the private key, an attacker would still
    need to steal the user's credentials, and would still be blocked by
    user-binding (biometric/PIN). Hence only the ISSUER signature truly needs to
    become PQ-secure (ML-DSA) plus a PQ-secure ZK proof.

A8. ISSUER + QUANTUM COMPUTER CAN RECOVER THE DEVICE PRIVATE KEY (SEMI-HONEST ISSUER)
    SOURCE: [YOU] - "Thesis/Thesis advance 02-05-2026.md"
    Description: If the issuer stores the user's device public key and possesses a
    quantum computer, it can invert ECDSA to recover the private key. A
    semi-honest issuer would then hold BOTH a valid attestation (it issued it) and
    a valid device-binding key pair - enabling impersonation or resale.
    Your mitigation: send the issuer a COMMITMENT (hash) of the device public key
    instead of the public key itself, so the issuer never sees the raw key.

A9. LOSS OF POST-QUANTUM UNLINKABILITY IF THE DEVICE PUBLIC KEY IS EVER EXPOSED
    SOURCE: [YOU] - "Thesis/Thesis advance 02-05-2026.md"
    Description: Even with a PQ-secure ZK proof, if the device public/private key
    is ever made visible (e.g. in the proof or to the issuer), a quantum adversary
    could use it as a persistent tracking handle WITHOUT even breaking it -
    destroying unlinkability. This drove your design choice to never reveal the
    device key and only commit to it.

(Self-found flaw in YOUR OWN proposed scheme - included for completeness)
A10. MALLEABLE COMMITMENT C = e1 XOR r ALLOWED FORGERY
    SOURCE: [YOU] - "Thesis/Thesis advance 02-05-2026.md"
    Description: In an earlier version of your PQ proof construction, building the
    commitment as C = e1 XOR r with both e1 and r private allowed a malicious
    prover to craft arbitrary r and e. You fixed this by switching to
    C = SHA256(e1 | r). (This is a flaw in your construction, not in EUDI itself.)

SOURCE PAPER: "Privacy Evaluation of the European Digital Identity Wallet's
Architecture and Reference Framework" (LINDDUN threat model on ARF v1.5)
Report note: Papers/Privacy Evaluation.../Report.md
--------------------------------------------------------------------------------
The paper first defines the persistent "quasi-identifiers" that enable tracking:
the user public key PKu, WUAid, EAArevoc_id, the SD-JWT fingerprint (salted-hash
digests, constant across presentations), OpenID4{VCI,VP} metadata (client_id,
access_token...), and the persistent certificates of ISO 18013-5 offline mode.

B1. LINKABILITY OF QUASI-IDENTIFIERS
    SOURCE: [PAPER]
    Presentations are signed with (and reveal) PKu; PKu and EAArevoc_id persist
    across sessions; salted hashes stay constant across presentations. Misactors
    (RPs, APs, WPs, supervisory bodies, external) can correlate repeat
    interactions and link them to attributable data.

B2. IDENTIFIABILITY RISK BETWEEN SERVICE PROVIDERS (RP + AP COLLUSION)
    SOURCE: [PAPER]
    A Relying Party can reconstruct full identity by cooperating with the
    Attestation Provider: APs collect linked pseudonyms, RPs/externals combine
    them with auxiliary data to re-identify the user.

B3. RE-IDENTIFICATION OF PSEUDONYMOUS IDENTITIES (LEGISLATIVE COMPULSION)
    SOURCE: [PAPER]
    Because pseudonym implementation is left to Member States, national
    legislative measures could force RPs/APs to disclose user information,
    violating privacy.

B4. NON-REPUDIATION OF WALLET-RELATED ACTIONS (NO PLAUSIBLE DENIABILITY)
    SOURCE: [PAPER]
    Digital signatures bind every wallet action to a single user. Once the
    verifying public key is correlated to an identifiable attribute, the user is
    uniquely identified and cannot deny involvement -> loss of plausible
    deniability.

B5. EXCESSIVE DATA DISCLOSURE TO PREVENT SERVICE EXCLUSION
    SOURCE: [PAPER]
    Nothing robustly defines which attributes are truly necessary. A Relying Party
    can demand more attributes than needed, pressuring the user (who may approve
    without understanding) and attempting identification.

B6. DATA FORMATS UNDERMINE DATA-PROTECTION PRINCIPLES
    SOURCE: [PAPER]
    The mandated standards/communication protocols do not accommodate
    unlinkability alongside signed data, making full eIDAS-envisioned data
    protection difficult/impossible. Some attestations (mDL, ID) are permanent and
    their data can be stored.

B7. DETECTABILITY OF WALLET ACTIONS VIA PASSIVE NETWORK MONITORING
    SOURCE: [PAPER]
    EUDIW communication runs over ordinary Internet networks and inherits their
    threat model. A passive Machine-in-the-Middle can monitor wallet usage
    without modifying traffic.

================================================================================
CROSS-CUTTING THEME
================================================================================
The single recurring root cause across nearly all privacy vulnerabilities
(A1, A2, B1-B6, B14, B30, B31) is the use of salted-hash selective disclosure +
persistent device/issuer signatures instead of Zero-Knowledge Proofs. Every
source - the ARF itself included - converges on ZKPs as the "only viable
mitigation". The orthogonal root cause for the security/availability issues
(A5, A6, A7, B19-B28, B31) is the dependency on a single, ECDSA-only,
quantum-vulnerable, poorly-available hardware Secure Element.
